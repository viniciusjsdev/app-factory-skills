# Backend Validation

- [ ] ORM classes use CamelCase.
- [ ] Models and matching Configurations use one module per entity and are exported correctly.
- [ ] DTOs, explicit mappers, and Controllers use scalable per-use-case modules without AutoMapper.
- [ ] Manifest version/status matches the implementation contract and contains no template values.
- [ ] Active Django settings use the manifest `ROOT_URLCONF`; inactive alternate URLconfs do not satisfy endpoint wiring.
- [ ] Environment URL variables match the manifest's exact URL, target service, and port.
- [ ] Every manifest endpoint uses its exact Controller/full effective path/method assignment across parent `include()` prefixes, has no uncontracted method, and has its exact required tests.
- [ ] Every manifest invariant has its exact positive/negative tests and command evidence.
- [ ] ORM access exists only in repositories.
- [ ] Services own business rules without database or HTTP access.
- [ ] Controllers use DTO-defined payloads and contain endpoint transport only.
- [ ] Mappers contain no authorization, business logic, ORM queries, or persistence.
- [ ] Every authored backend Python module, including tests and package `__init__.py` files, starts with a meaningful docstring explaining what the file does.
- [ ] Behavioral module docstrings identify responsibility, boundary, and relevant contract or `BR-###` references.
- [ ] Django-generated migrations were not edited to add docstrings.
- [ ] The required project-local layer skills exist and their hard boundaries were followed.
- [ ] Any added domain skill is product-specific, contract-backed, validated, and listed in `AGENTS.md`.
- [ ] Migrations were generated by Django commands.
- [ ] Layered tests, including mappers, and architecture scan passed or limitations are documented.
- [ ] Every required manifest validation ran; deferred required evidence prevents approval.
