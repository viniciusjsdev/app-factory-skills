# Django Backend Service Architect

## Purpose

Build or normalize Django backends for App Factory projects.

The backend must be clean, testable, API-first and prepared to connect to a frontend generated by Lovable and normalized by the frontend architecture skill.

## Core Principle

Django is the framework. The domain service layer owns business rules. Views/controllers only translate HTTP into service calls.

Do not put business rules inside views. Do not put business workflows inside serializers. Do not turn models into huge service objects. Do not let the frontend dictate unsafe backend structure.

## Expected Stack

Use this stack unless the repository already defines another compatible baseline:

- Python 3.12
- Django 5.x
- Django REST Framework
- PostgreSQL
- django-environ or python-decouple for environment variables
- psycopg
- gunicorn for production
- pytest or Django TestCase for tests
- ruff for linting when available

If the project already has `pyproject.toml`, preserve its package manager and tooling.

## Layer Responsibilities

### Models

Models represent persisted data.

Allowed:

- fields
- relationships
- constraints
- indexes
- simple row-level methods
- `__str__`
- model metadata

Not allowed:

- HTTP logic
- API formatting
- complex workflows
- external API calls
- frontend-specific fields unless persisted

### DTOs

DTOs represent service-layer input/output.

Use dataclasses or typed structures.

Allowed:

- service input objects
- service result objects
- typed domain payloads

Not allowed:

- database queries
- HTTP response formatting
- serializer validation logic duplicated unnecessarily

### Serializers

Serializers are API boundary objects.

Allowed:

- request validation
- response shape
- conversion between model/domain data and JSON-safe output

Not allowed:

- business workflows
- sending emails
- calling external APIs
- complex permission decisions
- hidden database mutations beyond simple serializer save when the endpoint is trivial

For non-trivial endpoints, serializers validate data and services perform the mutation.

### Services

Services own business actions.

Allowed:

- create/update/delete workflows
- transaction boundaries
- cross-model operations
- external integration orchestration
- domain validation
- raising domain exceptions
- calling selectors when useful

Not allowed:

- returning raw HTTP responses
- importing views
- rendering serializers
- knowing frontend routes
- swallowing exceptions silently

Services should be explicit:

```py
def create_project(*, data: CreateProjectDTO, actor: User) -> Project:
    ...
```

Use keyword-only arguments for clarity. Use `transaction.atomic()` for multi-step writes.

### Selectors

Selectors own read/query logic.

Allowed:

- queryset construction
- filters
- permissions-aware reads
- optimized prefetch/select_related
- list/detail retrieval

Not allowed:

- mutations
- business workflows
- HTTP formatting

### Views / Controllers

In Django/DRF, API views or viewsets are the controller layer.

Allowed:

- parse HTTP request
- call serializer validation
- call service or selector
- return response
- map domain exceptions to HTTP status

Not allowed:

- business workflows
- complex queries inline
- direct multi-model mutations
- hardcoded frontend assumptions

Views must stay thin.

## API Contract Workflow

Before implementing endpoints:

1. Read `docs/architecture/api-contract.md`.
2. Compare it with frontend service calls.
3. If missing, update the contract before coding.
4. Define endpoint path, method, request body, response body, status codes and errors.
5. Implement serializer/service/selector/view.
6. Add tests.
7. Update docs.

## Frontend Compatibility

The backend must serve the normalized frontend.

When replacing mock frontend services:

- preserve frontend DTO names when reasonable
- document mapping between frontend DTO and backend model
- avoid changing frontend behavior without updating the API contract
- include CORS settings for local development
- expose health endpoint `/api/health/`

## Required Baseline Endpoints

For most MVPs, create:

```txt
GET /api/health/
GET /api/me/
```

Then create domain endpoints according to the product.

## Authentication

Default for early MVP:

- prepare authentication structure
- do not overbuild auth unless requested
- support token/JWT integration later
- if Supabase Auth is used, prepare backend JWT validation boundaries but do not hardcode secrets

## Database

Use PostgreSQL.

If Supabase is used as managed Postgres:

- backend connects via `DATABASE_URL`
- do not use Supabase client directly in business logic unless the project requires Auth/Storage features
- Django models and migrations remain source of backend schema when Django owns the domain

If Supabase migrations are the source of truth, document the decision clearly.

Do not mix unmanaged schema ownership without documenting it.

## Testing

For each non-trivial feature, create tests for:

- service success
- service validation/domain errors
- selector filtering
- API happy path
- API error path
- permissions when applicable

Tests must be runnable locally and inside Docker.

## Validation Commands

After implementation, run the available equivalent of:

```bash
cd backend
python manage.py check
python manage.py makemigrations --check --dry-run
python manage.py migrate
python manage.py test
ruff check . || true
```

If using Docker:

```bash
docker compose build backend
docker compose run --rm backend python manage.py check
docker compose run --rm backend python manage.py test
```

Do not claim validation passed unless commands were actually executed.

## Architecture Scan

Run:

```bash
python scripts/scan-django-architecture.py
```

or the installed skill script path from the target backend.

The script should warn when:

- views contain too many lines
- views import models and mutate them directly in complex flows
- services import API views
- serializers contain suspicious business workflow terms
- apps lack tests
- endpoints are not documented
- `.env.example` is missing

The script is conservative. Warnings are acceptable; clear violations should fail.

## Definition of Done

Work is done only when:

- backend structure exists
- domain apps follow the service architecture
- models are clean
- services own business writes
- selectors own reads
- views/controllers are thin
- serializers/DTOs define API boundaries
- API contract is updated
- env examples exist
- Docker compatibility is preserved
- tests are created or updated
- validation commands were run or limitations were reported
