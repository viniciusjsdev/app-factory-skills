# Screen and Interaction Standard

For each route, modal, drawer, or materially distinct state, define:

- route or trigger
- objective
- allowed user roles
- layout regions and important components
- data displayed and its source contract
- entry and exit navigation
- loading, empty, error, populated, success, disabled, and confirmation states as applicable
- create, edit, delete, filter, search, sort, pagination, upload, export, print, and copy behavior when present
- desktop and mobile transformations
- accessibility requirements that are product-specific

Every visually presented action must have an outcome. Avoid dead buttons and unspecified controls.

Make mock data realistic, varied, domain-specific, and sufficient to demonstrate normal, boundary, empty, and error states. Avoid placeholder names such as `Teste 1`, `Usuário 1`, and lorem ipsum.

The primary journey must be executable from the screen map alone: entry point, intermediate actions, feedback, failure recovery, and successful completion must be clear.
